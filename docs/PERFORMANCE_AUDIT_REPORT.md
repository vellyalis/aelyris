# Aether Terminal パフォーマンス監査 & バグ調査レポート

**作成日:** 2026年5月3日
**ステータス:** 完了（読み取り専用調査に基づく）

---

## 1. エグゼクティブサマリー
Aether Terminalのコードベースを対象に、パフォーマンスのボトルネック、潜在的なバグ、およびアーキテクチャ上のリスクを調査しました。現行の実装は機能的に豊富ですが、**大規模なデータ出力時**および**長大な履歴（10,000行）保持時**において、システムのリソース消費が指数関数的に増大する設計上の課題が確認されました。

---

## 2. パフォーマンス・ボトルネックの分析

### 2.1 PTY通信のIPCオーバーヘッド (Critical)
- **該当箇所:** `src-tauri/src/ipc/commands.rs` -> `wire_terminal_streaming`
- **問題点:**
    - PTYからの生データを `tauri::AppHandle::emit` でフロントエンドへ送信する際、数値の配列としてJSONシリアライズされています。
    - 高スループット（例: `cat` による大容量ファイル出力、ビルドログ等）時に、シリアライズ/デシリアライズの負荷がCPUを圧迫し、`Lagged`（データドロップ）を誘発しています。
- **リスク:** ターミナル出力の欠落、UIのフリーズ、CPU使用率の異常高騰。

### 2.2 非効率なスクロールバック取得 (High)
- **該当箇所:** `src/shared/hooks/useScrollback.ts`
- **問題点:**
    - スクロール時、現在位置までの「全履歴」を一括で取得する設計になっています。
    - 履歴上限（10,000行）に達している場合、1回のスクロール操作で膨大な `CellSnapshot` オブジェクトがIPCを流れ、Reactの状態を更新します。
- **リスク:** メモリ使用量の激増、スクロール操作の著しい遅延。

### 2.3 メインループの同期処理負荷 (Medium)
- **該当箇所:** `src-tauri/src/ipc/commands.rs` 内の `loop`
- **問題点:**
    - データの受信ごとに `maybe_trigger_auto_repair`、ポートスキャン、ANSIベル検知を同期的に実行しています。
    - 出力が激しい場合、これらの処理時間が累積し、ターミナルエンジンのパース速度に追いつかなくなる可能性があります。

---

## 3. 潜在的なバグと信頼性のリスク

### 3.1 状態復元の不完全性
- **問題点:**
    - 権威あるイベントジャーナル（SQLite等）が存在せず、ターミナルの内容は alacritty のインメモリグリッドにのみ依存しています。
    - アプリのクラッシュや再起動時に、それまでのエージェントの思考プロセスやターミナル出力を完全に復元することができません。

### 3.2 メモリの二重管理
- **問題点:**
    - `alacritty_terminal` の内部グリッドと `OutputBuffer` の両方で、ほぼ同一のテキストデータを保持しています。
    - 複数タブを開くワークフローにおいて、メモリ消費が想定の2倍以上になる計算です。

### 3.3 IME（入力メソッド）の不整合
- **問題点:**
    - Canvasレンダリングにおいて、IMEの確定前文字列の描画が不安定であり、特定のウィンドウ操作（リサイズ、フォーカス遷移）で消失するリスクがあります。

---

## 4. 依存関係とリソース分析

- **重量級UIスタック:** Monaco Editor や Framer Motion の多用により、バイナリサイズと初期起動時のメモリフットプリントが増大しています。
- **IPCの限界:** Tauri 2.0の標準的な `emit` は、バイナリデータの高頻度転送には最適化されておらず、この部分がアプリ全体のボトルネックとなっています。

---

## 5. 推奨される改善策

1. **カスタムバイナリプロトコルの導入:** `tauri::ipc::Channel` を使用し、生バイトをシリアライズなしで直接フロントエンドにストリームする。
2. **スクロールバックのウィンドウ化:** `useScrollback` を修正し、ビューポート（例: 24〜50行分）のみを要求するページング方式に変更する。
3. **バックグラウンド・スキャナー:** 自動修復やポート検知などの解析処理を、メインのパースループから分離して非同期化（Worker thread化）する。
4. **イベントジャーナリングの実装:** すべての重要イベント（エージェントの思考、ツール実行）をSQLiteに書き込み、セッションの永続性を確保する。
