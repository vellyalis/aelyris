Audit goal (owner request): raise Aelyris to Apple/Linear-class **or better**
product experience — not "prettier", but a high-trust AI operating cockpit:
dense but calm, fast, native-feeling, evidence-first, understandable under
stress, safe for parallel AI work.

Method: static code audit by six parallel read-only investigations (shell/IA,
agent cockpit, terminal states, editor/SCM/settings, keyboard/a11y, verifier
infrastructure), cross-checked against the prior audits
(`UI_DENSITY_AUDIT_2026-07-03.md`, the 2026-07-03 terminal-sovereignty
scorecard) and the WU-UD-1 measured results. **No live app run this session**
— live evidence cited below is from the 2026-07-03 OS-level capture and the
WU-UD-1 Playwright measurements recorded in `ui-density-instructions.md`.
Populated-cockpit live inspection remains an OPERATOR GATE (§8).

## Standing constraints inherited from prior owner decisions

These are fixed. Any recommendation violating them is void:

1. **The transparent window is ABSOLUTE (owner law, reaffirmed 2026-07-05).**
   Per-pixel see-through (`DWMSBT_NONE`, no DWM material), glass alpha,
   backdrop-frost stay untouched — no phase of any work order may modify the
   DWM path, window background alpha, or backdrop-filter chains, and no new
   opaque full-bleed layer may be introduced. "Subtraction" may only target
   decorative layers stacked *on* glass, and only as an optional dial.
   Transparency cannot be verified via CDP; regressions are caught only by
   `verify:renderer:transparency` + OS-level operator eyes — treat both as
   standing gates.
2. **The 2026-07-03 layout redesign proposal was rejected by the owner**
   ("現状UI維持"). This audit does NOT re-propose terminal-sovereignty
   layout changes, mode-rail retirement, or chrome flattening. If the owner

3. **No bento revival.** `.bento-*` was measured as dead code; do not build on it.
4. **Never claim F12 = Monaco go-to-definition.** Confirmed this session:

   (`src/features/editor/lsp/registerProviders.ts:16-84`); no definition

5. Visible agents use interactive PTY paths; no `-p` for human-visible panes.
6. Alpha / not release-ready. Nothing here is a readiness claim; machine truth
   is regenerated by `pnpm verify:quality-score` / `pnpm verify:goal:safe`.

# PHASE 1 — Current UI/UX Reality

## 1. Verdict

**REVIEW.**

Not BLOCK: the substrate is genuinely strong — density work (WU-UD-1) landed
with measured results (2×2 dense: 86.4% of center is grid), the terminal has
real recovery machinery, approvals have a keystroke-safe latch, the merge
queue is genuinely git-backed, transparency is live-verified and excellent.

Not PASS: the product's own identity — *evidence-first, high-trust* — is
where the largest failures sit. Panes can be frozen-but-look-alive; a
"Blocked" merge verdict can come from regex-scraping a log; ownership and
blocked-reasons exist in the data model but are never rendered; and the
strongest UI proofs (real-layout Playwright, screenshots, CDP live checks)
are all **ungated** — CI verifies UI *source shape*, not rendered truth.

## 2. Executive diagnosis

1. **The craft is ahead of the trust.** Visual tokens, glass system, density
   modes, reduced-motion, and WCAG contrast math for terminal glyphs are real
   and better than most dev tools. The trust surfaces — is this pane alive,
   is this verdict evidence-backed, who owns what — are the weak half.
2. **Fake-alive pane is the single worst defect.** Sidecar stream drop retries
   silently up to ~10 strikes (`src-tauri/src/pty_sidecar.rs:947-984`);
   during that window the pane is frozen with a fully normal header. Session
   lifecycle (`paneLifecycleStates`, computed at
   `src/features/terminal/pane-tree/PaneTreeContainer.tsx:205,416-440`) is
   **never wired into `TerminalInfoBar`** — a dead shell renders identically
   to a live one.
3. **Multi-line paste auto-executes with no guard**
   (`src/features/terminal/IMEInputBar.tsx:550-595`): any `\r\n` paste is
   converted and submitted line-by-line immediately. Classic paste-injection
   footgun, directly against "safe for parallel AI work".
4. **Ownership is data-complete and UI-absent.** `AgentSession.owner`,
   `workspaceScope`, `writeSet` exist (`src/shared/types/agent.ts:108-113`)
   but no primary card renders them (`SessionCard.tsx:127-224`). Likewise
   `blockedReason`/`nextActor` (`agent.ts:118-121`) are rendered nowhere —
   the cockpit shows *that* something is blocked, never *why* or *who acts*.
5. **Merge-readiness has a two-class trust system.** `MergeQueuePanel` is
   evidence-backed (real `inspect_merge_worktree_branch` git states,
   `MergeQueuePanel.tsx:229-256`). `ReviewQueuePanel`'s Blocked/Ready badge
   is heuristic: `inferValidation` regex-scrapes agent logs for "error/pass"
   (`src/shared/lib/reviewQueue.ts:593-612`), coverage = filename matching
   (`:558-570`). One log line containing "error" flips a file to
   critical/blocked. The two look equally authoritative in the UI.
6. **The approval path — the most urgent action in the product — is
   mouse-only.** No keybinding exists to approve/deny
   (`useKeyboardShortcuts.ts` has none; `DecisionInboxPanel.tsx:168-202`).
7. **Navigation state is not owned.** `productMode`/`rightRailMode` are
   `App.tsx` local `useState` (`App.tsx:491-493`), never persisted — every
   reload resets to Terminal/command. Layout state is split across three
   owners (Zustand store, App.tsx locals, workspaceProfile).
8. **Two of eight mode-rail entries are fake modes.** Settings and History
   route to dialogs (`rightRailModel.tsx:1113-1122`) while still setting
   `productMode`, so the rail highlight diverges from what is on screen.
9. **The right-rail `command` mode buries its own actionables** under ~1200
   lines of inspector hero/edge-score/goal-track JSX (`App.tsx:3439-4624`)
   before the first actionable widget (`:4635`).
10. **Ghost layers persist:** `features/sidebar/Sidebar.tsx` + store field
    `sidebarSection` are dead (only consumer never imported); the entire Rust
    keybinding config (`src-tauri/src/config/keybindings.rs`) is unwired —
    its documented `Ctrl+Shift+H/V` splits do not fire, and the Settings
    "Keyboard Shortcuts" list is a hand-maintained literal that can drift
    (`Settings.tsx:1218-1247`).
11. **UI proof exists but is ungated.** `e2e/visual-qa-layout.spec.ts` (real
    layout math, keyboard traversal, DPR×density matrix) and screenshot
    baselines run only manually; CI has no Playwright job. `verify:quality-score`
    greps the *test source text* (`score-release-quality.mjs:432,2035-2037`) —
    a rendered-layout regression cannot move the score. The only gated UI
    checks are static CSS/JSX greps (glass-legibility, right-rail density).
12. **A11y floor is breached systemically but fixably:** 9-10px text tokens
    across ~140 CSS files, sub-24px interactive targets, error toasts not
    escalated to assertive; against that, aria breadth (683 uses / 98 files),
    reduced-motion, and Radix focus traps are genuinely good.

# PHASE 3 — UX principles for Aelyris

Product-specific, each with the anti-pattern it bans:

1. **Evidence over optimism.** A green state must be backed by a machine
   check; an inferred state must look inferred. *Implication:* verdict chips
   carry a provenance tag (verified/inferred/stale); "Blocked/Ready" without
   drill-down is forbidden. *Anti-pattern:* `reviewQueue.ts` regex verdicts
   styled identically to git-backed ones. *Surfaces:* ReviewQueuePanel,
   right-rail advisor.
2. **No fake-alive surfaces.** Anything that can die (PTY, sidecar stream,
   agent, LSP) must render its liveness. *Implication:* lifecycle dot in
   every pane header; "reconnecting…" is a first-class state. *Anti-pattern:*
   silent WebSocket retry behind a normal-looking pane. *Surfaces:*
   TerminalInfoBar, agent chip, FleetHud.
3. **One owner per state.** Every piece of layout/nav state has exactly one
   store and survives reload. *Anti-pattern:* productMode in App.tsx useState,
   density in workspaceProfile, panels in Zustand. *Surfaces:* app shell.
4. **The dangerous action is the easiest to do safely.** Approve/deny,
   stop-agent, paste-run get keyboard paths, latches, and previews before
   anything decorative gets polish. *Anti-pattern:* mouse-only approvals
   while Alt+1..8 mode switching exists. *Surfaces:* DecisionInbox, IMEInputBar.
5. **Visible agents over hidden automation.** (Standing) 1 agent = 1 visible
   PTY pane; ownership rendered per lane. *Anti-pattern:* `owner`/`writeSet`
   fields that exist only in TypeScript. *Surfaces:* SessionCard, lane cards.
6. **Dense but calm.** Density comes from removing chrome (UD-1), never from
   shrinking text below the 11px floor. *Anti-pattern:* 9px telemetry tokens
   spreading into labels. *Surfaces:* all chrome strips.
7. **Native terminal confidence.** The grid never lies: time-travel overlays,
   degraded durability, and backpressure are visibly distinct from live
   output. *Anti-pattern:* snapshot overlay whose glyphs look live except a
   small pill (`TimelineBar.tsx:94-96`). *Surfaces:* NativeTerminalArea, TimelineBar.
8. **Progressive disclosure for power.** Primary work first; analytics walls
   collapse. *Anti-pattern:* 1200 lines of hero JSX above the Run-mode
   actionables. *Surfaces:* right rail command mode.
9. **Proof before merge.** The merge surface only ever shows git-backed
   states (MergeQueuePanel is the template); Proofbook UI, when built, binds
   PB-1..PB-4 runtime ledgers, never mocks (PB spec stop-condition stands).
10. **Remote observation is safe, remote control is gated.** (Standing D2R
    principle; UI must render lease/principal state when that lands.)
11. **No unwired capability.** A rendered control or documented shortcut that
    does nothing is a bug of the same class as fake state. *Anti-pattern:*
    keybindings.rs defaults, GhostDiffPanel stub click, static Settings
    shortcut list. *Surfaces:* Settings, GhostDiff, help.

# PHASE 4 — Target UX architecture

Kept deliberately close to the current shell (layout redesign is owner-rejected;
this is a hardening architecture, not a reshuffle):

1. **App shell.** Current regions stay (rail / left / center / right / tabs /
   status). Changes are ownership, not geometry: a single persisted
   `workspaceView` slice { productMode, rightRailMode, focusWidget } in the
   Zustand store; mode rail entries that open dialogs are marked as launchers
   (no productMode mutation); F6/Shift+F6 rotates regions.
2. **Agent cockpit.** `useAgentFleet()` becomes the only fleet source
   (FleetHud migrates off `useTaskGraph` join). Lane cards gain an ownership
   line (`owner · scope · writeSet count`) and a blocker line
   (`blockedReason → nextActor`). Conductor stays the handoff view; Parallel
   stays the lane view; HUD stays ambient — same data, three projections.
3. **Terminal/pane model.** Pane header = identity (existing) + **liveness
   dot** (live / reconnecting / exited / detached) + durability marker when
   `degraded`. Sidecar reconnect becomes an emitted event rendered as a
   badge. Paste of >1 line opens preview-confirm (opt-out via settings).
   Time-travel overlay gets a stronger "viewing the past" treatment than the
   current pill — but any treatment must preserve window see-through (no
   opaque scrim; prefer border/desaturation on glyphs) and passes the
   operator transparency gate before claim.
4. **Proofbook UX (future, gated).** When PB UI unlocks: run setup binds a
   validated PB (PB-1), step status streams from the runner ledger (PB-2),
   evidence links open artifacts, settlement mirrors PB-4 rules
   (done-signal / final report / required artifacts / reviewer proof). The
   ReviewQueue provenance-trace interaction (chips → open evidence) is the
   interaction template. Build nothing before its backend gate — the PB spec
   stop-condition is binding.
5. **Remote continuity UX (future, gated).** Read-only monitor first; approval
   actions render fingerprint + lease state; SSH attach appears as a pane
   with an explicit remote-principal marker. Follows D2R detailed design; no
   UI before RC work units land.
6. **Settings model.** Settings keeps: appearance/material, terminal config,
   updates, shell integration. Moves out: read-only shortcut list becomes a
   generated Help surface fed by the same table that registers bindings;
   per-workflow toggles (Open Files With, ghost live mode) become palette
   commands. The material/wallpaper editor is a candidate standalone panel
   (defer).

Execution granularity, file lists, tests, and gates are in

   Q0 gate-first trust verifier → Q1 liveness → Q2 reconnect events (small
   Rust) → Q3 paste guard → Q4 ownership/blockers on cards → Q5 keyboard

   `PasteGuardDialog` (IMEInputBar), `OwnershipLine`/`BlockerLine`

# PHASE 6 — Adversarial review of this plan

**BLOCKING risks:**
- **Q3 paste guard can break IME composition flows** — the same component
  owns both. Mitigation: guard only the paste event path, never composition;
  OPERATOR GATE on real Japanese IME (same discipline as UD-1/U1).
- **Q2 touches `src-tauri` stream code** — a bug here can regress PTY
  streaming for every pane. Mitigation: emission-only change (no protocol,
  no state machine edits), `cargo test --lib`, and the existing
  chunked-OSC/live verifiers untouched-green.
- **Q10 flipping Playwright to blocking CI on a Windows runner can flake.**
  Mitigation: land as continue-on-error with artifact upload; flipping to
  blocking is an explicit later OPERATOR decision after observed stability.

**REVIEW risks:**
- Q4/Q6 depend on data quality: if `owner`/`writeSet` are rarely populated at
  runtime, rendering them adds empty chrome. Phase includes a
  render-only-when-present rule.
- Q8 persisting nav state may surprise users who rely on reset-to-terminal.
  Persist per workspace profile, and zen stays transient.
- WU-FA-1 (`feat/wu-fa-1-fleet-api`, unmerged) also edits `TerminalInfoBar`
  (`%N` prefix). Do not run both work orders concurrently; whichever lands
  second rebases and re-runs the density + trust verifiers.
- `App.tsx` is a 5148-line hotspot with three unmerged branches touching it
  historically — keep Q-phase diffs surgical, no opportunistic refactors.

**Where the Apple/Linear comparison misleads:**
- Apple-class "calm" tempts hiding state; in a cockpit, hidden state is the
  enemy. Aelyris must show *more* operational truth than Apple would.
- Linear's minimal chrome works because its objects are homogeneous (issues).
  Terminal panes, agents, diffs, and proofs are heterogeneous — uniform card
  chrome would flatten meaningful differences.

**Do not copy from Apple:** hiding scrollbars/affordances, marketing-grade
motion, ultra-low-contrast secondary text on glass.
**Do not copy from Linear:** single-accent monochrome flattening (state
colors carry meaning here), hiding data behind hover, web-app chrome.
**Where Aelyris must be better:** liveness/trust states (neither Apple nor
Linear runs PTYs), evidence drill-down, parallel-lane comprehension,
paste/approval safety.

**Top 10 by quality-per-effort:** Q1, Q7-toast, Q4, Q5, Q6, Q3, Q2, Q0,
Q9, Q8 (Q10 is high value but external-flake-exposed).
